Contiguous & Non-Contiguous Tensors

How tensors live in memory — strides, views, and why it matters

Memory is a flat 1D strip of addresses. A tensor is just a recipe for reading that strip as a multi-dimensional object. The recipe is encoded in three numbers: shape, strides, and a storage offset.

Key concepts

Strides
Number of memory cells to skip to advance one step in each dimension. A 3×4 contiguous tensor has strides [4, 1].
Contiguous
Elements are laid out sequentially in memory — iterating the tensor walks memory in a straight line. Fast for the CPU/GPU cache.
Non-contiguous
Strides cause memory access to jump around. Created by .T, permute(), slicing with a step, expand(), etc.
.view()
Requires contiguous memory — will raise an error otherwise.
.reshape()
Works on non-contiguous tensors but may silently copy data.
.contiguous()
Forces a copy into a fresh row-major layout. Always contiguous after this.

Why this matters in LLMs

Attention score computation involves a lot of transposes and permutes. After a .transpose(), the tensor is non-contiguous. If you then call .view() to reshape, PyTorch will throw a RuntimeError. The fix you'll see everywhere in transformer code:

x = x.transpose(1, 2)   # now non-contiguous
x = x.contiguous()      # copies to fresh layout
x = x.view(B, T, C)     # safe to view now
.contiguous() triggers a memory allocation and copy — it's not free. In tight training loops, it can be a hidden bottleneck. Profile before adding it blindly.
