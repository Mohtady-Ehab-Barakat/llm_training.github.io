Attention efficiency

GQA & MQA

Multi-Query · Grouped Query — reducing the KV cache without losing quality

The problem — KV cache memory at inference

In standard Multi-Head Attention with H heads, the KV cache grows with every generated token:

KV cache = 2 × L × T × H × head_dim × bytes
         = 2 × layers × seq_len × heads × head_dim × dtype

# LLaMA-7B at T=4096, fp16:
# 2 × 32 × 4096 × 32 × 128 × 2 = 2 GB (cache alone)
# At T=32768: 16 GB — more than the model weights

At inference, generating each new token requires reading the entire KV cache from GPU memory. This is a memory bandwidth bottleneck, not a compute one. Reducing the KV cache directly enables: longer context windows in the same GPU memory, higher batch sizes, and faster token generation.

MHA vs MQA vs GQA — head organisation

Multi-Query Attention (MQA)

MQA (Shazeer, 2019) takes the most aggressive approach: all H query heads share a single K and a single V projection:

Q_h = X @ Wq_h   # for h in 0..H  — H separate Q projections
K   = X @ Wk     # ONE shared K projection
V   = X @ Wv     # ONE shared V projection

# Each head h uses its own Q but the shared K and V:
scores_h = Q_h @ K.T / sqrt(head_dim)
output_h = softmax(scores_h) @ V

KV cache shrinks from 2×H to 2×1 — an H× reduction. For H=32 that's 32× smaller. The cost: all H heads are forced to search the same key-value database. They can still ask different questions (different Wq_h), but they're all searching the same "library." Head specialisation is limited, and quality drops — especially on longer sequences and larger models.

Grouped Query Attention (GQA)

GQA (Ainslie et al., 2023) interpolates between MHA and MQA by dividing the H query heads into G groups. All heads within a group share one K and one V:

group_size = H / G    # Q heads per group

# For group g (heads g*gs to (g+1)*gs - 1):
K_g = X @ Wk_g        # one K projection per group
V_g = X @ Wv_g        # one V projection per group

# For each head h in group g:
Q_h     = X @ Wq_h    # still independent per head
scores_h = Q_h @ K_g.T / sqrt(head_dim)
output_h = softmax(scores_h) @ V_g

# KV cache scales with G, not H:
KV cache = 2 × L × T × G × head_dim × bytes
The key finding from the GQA paper: G = H/8 recovers nearly all of MHA quality while achieving near-MQA cache efficiency. This makes GQA strictly better than MQA in almost all cases — slightly more cache cost for significantly better quality. G=8 is the most common choice regardless of H.

Why G=8 is usually enough

Within a group of H/G query heads sharing one KV, the heads can still ask very different questions — they just search the same key space. Think of multiple people searching the same library catalogue using different search terms. The catalogue (K) and books (V) are shared, but each person finds what they need through their own query. As long as the group isn't too large, the shared KV space is expressive enough to support diverse attention patterns.

When G=1 (MQA), all 32 or 64 heads share one catalogue — too limiting at scale. When G=H/8, groups of 4–8 heads share a catalogue — diverse enough that each group can support distinct patterns within it.

Parameter count difference

# For H=32, G=8, D=4096, head_dim=128:
MHA  Wk + Wv: 2 × 4096 × (32×128) = 2 × 4096 × 4096  = 33.6M / layer
MQA  Wk + Wv: 2 × 4096 × (1×128)  = 2 × 4096 × 128   =  1.0M / layer  (32× less)
GQA  Wk + Wv: 2 × 4096 × (8×128)  = 2 × 4096 × 1024  =  8.4M / layer  (4× less)

# LLaMA models compensate by increasing FFN size.

Converting MHA → GQA

The GQA paper showed you can convert a pretrained MHA model to GQA with uptrain — fine-tuning on a small fraction of the original training data. The conversion groups existing MHA heads, mean-pools the K and V weight matrices within each group, then fine-tunes to recover quality. This was used to create GQA versions of originally-MHA models without full retraining.

MHA
Multi-Head Attention. H Q heads, H K heads, H V heads. Maximum quality, maximum cache. The original design.
MQA
Multi-Query Attention. H Q heads, 1 shared K, 1 shared V. H× cache reduction, some quality loss.
GQA
Grouped Query Attention. H Q heads, G shared KV groups (1 ≤ G ≤ H). H/G× cache reduction, near-MHA quality at G=H/8.
KV cache bandwidth
The main inference bottleneck at long context. Every generated token requires reading the full cache from GPU memory. Smaller cache = faster generation.
G = H/8
The empirically validated sweet spot. Used in LLaMA-2-70B, LLaMA-3, Mistral, Falcon-40B. Nearly recovers MHA quality.
Uptrain
Converting a pretrained MHA model to GQA by mean-pooling K/V weights and fine-tuning briefly. Avoids full retraining.
The practical summary: GQA with G=8 is now the default for any model that cares about serving efficiency. It costs almost nothing in quality, reduces the KV cache by H/8× (4–8× for typical H), and enables much longer contexts or much higher batch sizes in the same GPU memory. If you're designing a new LLM in 2024+, there's almost no reason to use full MHA.
