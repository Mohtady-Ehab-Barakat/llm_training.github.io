Attention

Keys & Values — Q, K, V

Where they come from · what they are · the full computation · multi-head · KV cache · GQA

The problem attention solves

Each token starts with an embedding vector — D numbers encoding its meaning. But meaning is context-dependent: "bank" in "river bank" vs "savings bank" should produce different representations. After the embedding lookup, every "bank" token has the same vector regardless of context. Attention lets each token update its representation by reading information from other tokens in the sequence.

The database analogy

The best mental model for attention is a soft, differentiable database lookup. In a real database you have a query, a set of keys to match against, and values to retrieve. The match is hard — one key wins. Attention does the same thing but softly: every key gets a score, softmax converts scores into weights summing to 1, and you get back a weighted blend of all values:

# Hard database lookup
query = "find me animal records"
keys  = ["animals", "weather", "finance"]
→ return values["animals"]   # 100% one winner

# Soft attention lookup
→ return 0.72×V_animals + 0.18×V_weather + 0.10×V_finance

Q (Query), K (Key), V (Value) are the neural network version of this. Q is what you're looking for. K is what each token advertises. V is what each token contributes when selected.

Where Q, K, V actually come from

The input to the attention block is the residual stream: a matrix X of shape [T, D] — one D-dimensional vector per token. This same X produces all three of Q, K, and V by multiplying through three separate learned weight matrices:

Q = X · Wq      # [T, D] × [D, Dh] = [T, Dh]   "what do I need?"
K = X · Wk      # [T, D] × [D, Dh] = [T, Dh]   "what do I advertise?"
V = X · Wv      # [T, D] × [D, Dh] = [T, Dh]   "what do I contribute?"

Wq, Wk, Wv are weight matrices with their own learned values, updated during training. Every token is simultaneously a querier (via its Q), a key-provider (via its K), and a value-provider (via its V). When token i attends to token j, it uses its own Q to search, matches against j's K, and takes j's V as the payload.

What K is — and why it exists separately from V

K is the "advertisement" each token broadcasts. When Wk projects X into key space, it learns to answer: "if another token is searching, what label should I put on myself?" The key for a verb might encode "I am an action." The key for a pronoun might encode "I am a reference needing resolution."

K is never used as content — it only participates in the scoring computation Q·Kᵀ. Its sole job is to be findable. Whether you find it depends on how well the Q vector (the searcher) aligns with the K vector (the label).

Why split K and V? The label a token advertises (K) and the content it contributes (V) don't need to encode the same thing. A token might advertise "I am a subject noun" via K while contributing rich semantic information via V. The model learns to search with one representation and retrieve with another — like a library where the card catalogue keywords (K) differ from the book's contents (V).

What V is — the actual payload

V is the content that gets collected when a token is attended to. After softmax gives attention weights, the output for token i is the weighted sum of all V vectors:

output[i] = Σⱼ  weight[i,j] × V[j]

Token i's output is a blend of every token's value vector, weighted by how well each key matched the query. High attention weight on token j means j's V vector contributes heavily to i's new representation. This is how contextual information flows — "cat" can absorb information from "sat" and "mat" to become a context-aware representation of a cat-sitting-on-mat.

The full computation — all steps

# All attention operations for one head:

Q = X @ Wq                         # [T, Dh]  project to query space
K = X @ Wk                         # [T, Dh]  project to key space
V = X @ Wv                         # [T, Dh]  project to value space

scores = Q @ K.T / sqrt(Dh)        # [T, T]   score every pair
# causal mask: scores[i,j] = -inf for j > i  (decoder-only)

weights = softmax(scores, dim=-1)  # [T, T]   normalise to sum=1

output = weights @ V               # [T, Dh]  weighted value blend
output = output @ Wo               # [T, D]   project back to D

Why divide by √D_head?

The dot product Q·K sums D_head multiplications. As D_head grows, the variance of the dot products grows proportionally — large scores make softmax peak sharply toward one token, destroying gradient flow (softmax gradient near saturation ≈ 0). Dividing by √D_head keeps scores in a sensible variance range. For D_head = 64, you divide by 8. For D_head = 128, you divide by ~11.3.

Multi-head attention — running this H times

A single head can only attend to one type of relationship at a time. Multi-head attention runs H independent attention operations in parallel, each with its own Wq, Wk, Wv:

head_dim = D / H   # D split evenly across heads

for h in range(H):
    Qh = X @ Wq_h                              # [T, head_dim]
    Kh = X @ Wk_h                              # [T, head_dim]
    Vh = X @ Wv_h                              # [T, head_dim]
    head_h = softmax(Qh @ Kh.T / √head_dim) @ Vh

output = concat(head_0, ..., head_H-1) @ Wo   # [T, D]

Head 0 might track subject-verb agreement. Head 1 might track pronoun-antecedent links. Head 7 might track semantic similarity. The model learns to specialise each head through gradient descent. Wo blends all heads back into the unified D-dimensional residual stream.

The KV cache — why K and V are stored during inference

During autoregressive generation (one token at a time), at each new step you already have K and V for all past tokens — their source token representations haven't changed. Recomputing them is wasteful. The KV cache stores K and V for every past token, every layer:

Cache size = 2 × L × T × D × bytes_per_value
             ↑      ↑   ↑   ↑
           K&V   layers seq width

# LLaMA-7B at T=4096, fp16:
# 2 × 32 × 4096 × 4096 × 2 = ~2 GB

At each new token: compute only the new token's K and V, append them to the cache, then compute attention between the new Q and all cached Ks. This reduces attention from O(T²) per step to O(T) — the main reason modern LLM inference is fast. The Q of the new token is computed fresh every step and never cached (used once and discarded).

Q is never cached. K and V always are. Q is the "searcher" — it changes every step as the new token's needs change. K and V are the "database" — past tokens' advertisements and payloads are fixed once generated. This asymmetry is why the operation is called a Key-Value cache, not a QKV cache.

GQA and MQA — sharing K and V across heads

Standard MHA has H separate Wq, Wk, Wv per head. The KV cache grows linearly with H. Multi-Query Attention (MQA) shares one single K and V across all H query heads. Grouped Query Attention (GQA) uses G groups of shared KV for H Q heads, where G < H:

MHA (standard): H separate (Wq, Wk, Wv) — KV cache ∝ H
MQA:            H Wq, but only 1 shared (Wk, Wv) — KV cache ÷ H
GQA (G groups): H Wq, G shared (Wk, Wv) — KV cache ÷ (H/G)

# LLaMA-70B: H=64 Q heads, G=8 KV groups
# KV cache is 8× smaller than standard MHA
# Quality loss: nearly zero

LLaMA-2-70B, Mistral-7B, and most modern efficient LLMs use GQA. It's the primary technique for fitting long contexts in GPU memory without sacrificing model quality.

Q — Query
What this token is looking for. Computed as X·Wq. Used only in Q·Kᵀ scoring. Never cached.
K — Key
What this token advertises. Computed as X·Wk. Used only in Q·Kᵀ scoring. Always cached after first computation.
V — Value
What this token contributes when attended to. Computed as X·Wv. Collected as a weighted sum. Always cached.
Wq, Wk, Wv
Three independent learned [D, D_head] weight matrices. Their job is to project the same X into three different useful spaces.
Wo
Output projection [D_head × H, D]. Blends all heads back to D and writes into the residual stream.
Scaled dot-product
Q·Kᵀ / √D_head. Scaling prevents softmax saturation at large head dimensions.
Causal mask
Sets future positions to −∞ before softmax so each token can only attend to past tokens. Used in all decoder-only LLMs (GPT, LLaMA).
KV cache
Stored K and V for all past tokens, all layers. Enables O(T) per-step inference instead of O(T²). Size = 2·L·T·D·dtype_bytes.
MHA
Multi-Head Attention — H independent heads, each with own Wq, Wk, Wv.
MQA
Multi-Query Attention — H Q heads sharing one K, V. Shrinks KV cache by H×.
GQA
Grouped Query Attention — H Q heads sharing G KV groups (G < H). Used in LLaMA-2-70B, Mistral.
